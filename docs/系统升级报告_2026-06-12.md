## 二、逐项对比

### 2.1 SSE 长连接稳定性

| 对比维度 | 升级前（旧系统） | 升级后（新系统） |
|----------|----------------|----------------|
| Gunicorn timeout 设置 | `--timeout 300`（5 分钟） | `--timeout 3600`（1 小时）+ 应用层 55 秒主动断开 |
| SSE 连接行为 | 每 ~5 分钟被 Gunicorn 强制 kill，Nginx 报 `upstream prematurely closed connection` | 每 55 秒优雅断开，浏览器 `EventSource` 自动重连（用户无感知） |
| Nginx 错误日志 | 持续出现 SSE 断连错误 | 零条 SSE 断连错误 ✅ |
| 实时消息推送 | 不可靠，用户经常看不到新消息通知 | 可靠，消息实时到达 |

**技术原理**：
- 旧系统：`while True` 永久阻塞，Gunicorn 认为 worker 卡死，触发 timeout 杀进程
- 新系统：SSE 生成器每 55 秒主动 `break`，浏览器自发重连，Gunicorn timeout 永不被触发

### 2.6 慢请求监控（新增功能）

| 对比维度 | 升级前 | 升级后 |
|----------|--------|--------|
| 慢请求感知 | 无（用户投诉才知道） | 有 ✅ |
| 监控方式 | — | Flask 中间件：`before_request` 记录开始时间，`after_request` 计算耗时 |
| 告警阈值 | — | > 1.0s：WARNING 日志；> 3.0s：ERROR 日志 |
| 日志输出 | — | `journalctl -u grade7-new` 可直接查看 |

**代码示例**（已写入 `app.py`）：
```python
@app.before_request
def _start_timer():
    g._start_time = time.time()

@app.after_request
def _log_slow_request(response):
    if hasattr(g, "_start_time"):
        duration = time.time() - g._start_time
        if duration > 1.0:
            app.logger.warning(f"[慢请求] {request.method} {request.path} 耗时 {duration:.2f}s")
    return response
```
